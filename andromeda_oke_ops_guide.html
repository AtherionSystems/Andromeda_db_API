Andromeda OKE — Ops Guide

OKE cost management · Mexico Central (Querétaro) · Andromeda project

Stop the bleeding first. You are currently being billed ~$5/hr for a Load Balancer + OKE control plane with zero useful work being done.
Immediate actions
1
Delete the Load Balancer highest priority
Go to Networking → Load Balancers in OCI Console (same region: Mexico Central Querétaro). Find any LB whose name contains andromedaoke or whose tags link to the cluster. Delete it.

Alternatively from Cloud Shell:
oci lb load-balancer list --compartment-id <your-compartment-ocid> --all # Find the OCID of the LB tied to your cluster, then: oci lb load-balancer delete --load-balancer-id <lb-ocid> --force
Expected output
LB disappears from the list within 1–2 min. Cost line "Load Balancer" drops to 0.
  • If stuck in "Deleting" state: the LB is still referenced by a K8s Service. Run kubectl delete svc --all -n andromeda first.
2
Terminate the stopped compute instances
In Instances (Image 4), the three oke-c5zeswx6pna-* nodes show "Terminated" but their boot volumes may still exist. The three oke-cwhpkfiezpq-* nodes are "Stopped" — these are definitely still billing for compute + boot volume.

Go to Compute → Instances, select all 6, click Actions → Terminate. Check the box "Permanently delete the attached boot volume" for each.
Expected output
All instances move to Terminated state. The boot volumes list under Block Storage should show empty or only your Autonomous DB volume.
  • If OKE re-provisions nodes automatically: your node pool desired count is not 0. Go to Node Pools → Edit → set Size to 0 before terminating.
3
Set node pool size to 0
In Containers → Clusters → andromedaoke → Node Pools → pool1 → Edit. Set "Number of nodes" to 0. Save.
Expected output
Pool shows 0 total worker nodes. No new instances will be auto-provisioned.
4
Check boot volumes separately
Go to Block Storage → Boot Volumes. Filter by compartment reacttodo. Delete any orphaned volumes not attached to a running instance.
Expected output
Boot Volumes list is empty or contains only your DB-related volumes.
After these 4 steps, your only remaining cost should be the OKE control plane (~$0.10/hr, unavoidable while the cluster exists) and the Autonomous DB (typically near-zero at low load). The $3–4/hr Load Balancer line will be gone.
Run this at the end of every work session. The full teardown takes about 5 minutes and reduces cost to near zero.
Option A — Console (recommended for the team)
1
Delete Kubernetes services that provision cloud resources
From Cloud Shell, before anything else:
kubectl delete svc --all -n andromeda kubectl delete ingress --all -n andromeda
Expected
service "andromeda-service" deleted (or similar). This releases the Load Balancer in OCI.
  • If namespace does not exist yet: skip this step.
  • If kubectl is not configured: run oci ce cluster create-kubeconfig --cluster-id <ocid> --region mx-queretaro-1 first.
2
Scale node pool to 0
Containers → Clusters → andromedaoke → Node Pools → pool1 → Edit → Number of nodes: 0 → Save

Or via CLI:
oci ce node-pool update \ --node-pool-id <node-pool-ocid> \ --size 0
Expected
Instances begin terminating (1–3 min). Node pool shows 0 total workers.
  • Instances stuck in "Stopping": wait 5 min, then manually terminate from Compute → Instances.
3
Confirm Load Balancer is gone
Networking → Load Balancers. The LB provisioned by your K8s LoadBalancer Service should be deleting or gone after step 1.
Expected
Empty list or only pre-existing LBs unrelated to Andromeda.
  • LB still present: delete it manually (see Right Now tab, step 1).
4
Delete orphaned boot volumes
Block Storage → Boot Volumes, filter compartment reacttodo. Delete any volume not attached to a running instance.
5
Verify in Cost & Usage
Wait 5–10 min, then check Billing → Cost Analysis. The "Container Engine for Kubernetes" bar should drop sharply. Load Balancer should be 0.
Option B — Full cluster delete (use when no work for 2+ days)
Only do this if no one will need the cluster for more than 2 days. Reprovisioning takes ~15 min.
1
Delete kubectl services first (same as Option A step 1)
kubectl delete svc --all -n andromeda kubectl delete ingress --all -n andromeda
2
Delete the cluster
Containers → Clusters → andromedaoke → Delete. Confirm. OCI will terminate all node pool instances automatically.
Expected
Cluster disappears from the list. All node instances terminate. LB is released.
  • VCN will persist — that is fine, it costs nothing meaningful.
  • Autonomous DB is not deleted by this action — your data is safe.
If you used Option A teardown (cluster kept, nodes at 0): only do steps 3–5. If you deleted the cluster entirely: do all steps.
Full bring-up (~15 min from scratch, ~5 min from Option A)
1
Recreate the cluster only if fully deleted
Containers → Clusters → Create cluster → Quick Create
Settings to use:
Name: andromedaoke Compartment: reacttodo Kubernetes ver: v1.34.2 (same as before) Shape: VM.Standard.E3.Flex OCPUs: 1 per node Memory: 6 GB per node Node count: 1 (scale up if needed) VCN: Create new or reuse oke-vcn-quick-andromedaoke-281194dc7
Expected
Cluster status: Active within 10–12 min.
2
Configure kubectl only if fully deleted
In Cloud Shell (or local terminal with OCI CLI):
oci ce cluster create-kubeconfig \ --cluster-id <new-cluster-ocid> \ --region mx-queretaro-1 \ --token-version 2.0.0 \ --kube-endpoint PUBLIC_ENDPOINT kubectl get nodes
Expected
NAME ... STATUS Ready for each node.
  • Nodes show NotReady: wait 2–3 more min for kubelet startup.
  • "No resources found": kubeconfig is pointing at wrong cluster — re-run the create-kubeconfig command.
3
Scale node pool back to 1 Option A path
Node Pools → pool1 → Edit → Number of nodes: 1 → Save
Wait for instance to become Running and node to become Ready.
Expected
kubectl get nodes shows 1 node Ready.
4
Apply your Kubernetes manifests
From your repo root:
kubectl apply -f k8s/namespace.yaml kubectl apply -f k8s/configmap.yaml kubectl apply -f k8s/secret.yaml kubectl apply -f k8s/deployment.yaml kubectl apply -f k8s/service.yaml
Or if you have a single manifest file:
kubectl apply -f k8s/andromeda.yaml
Expected
kubectl get pods -n andromeda shows pod Running (not Pending). Takes 2–5 min for image pull.
  • Pod stuck Pending: node not Ready yet — wait. Check with kubectl describe pod <name> -n andromeda.
  • ImagePullBackOff: check your OCIR secret is present and image tag is correct.
  • CrashLoopBackOff: application startup error — check kubectl logs <pod> -n andromeda.
5
Confirm Load Balancer is provisioned and get external IP
kubectl get svc -n andromeda
Expected
Service shows EXTERNAL-IP (not <pending>). Takes 2–4 min after kubectl apply.
  • EXTERNAL-IP stuck pending: LB is still provisioning in OCI — wait and retry.
6
Update the team with the new external IP
Each time the LB is reprovisioned it may get a new IP. Post the new IP in your team channel immediately after step 5.
